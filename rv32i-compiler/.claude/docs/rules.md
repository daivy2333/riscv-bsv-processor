# 项目编码规范与行为约束

> 最终更新：2026-05-12
> 适用于：rv32i-compiler

---

## 一、Karpathy Guidelines（行为约束）

### 1. Think Before Coding
**不假设。不隐藏困惑。暴露权衡。**

- 实现前明确陈述假设，不确定就问
- 多种解读存在时，全部呈现，不 silently 选择
- 更简单的方法存在时说，必要时 push back
- 不清楚时立即 STOP，命名困惑点并询问

### 2. Simplicity First
**最小代码解决问题。无投机性功能。**

- 不添加未被要求的功能
- 单次使用代码不抽象
- 未要求的"灵活性"或"可配置性"不加
- 不可能场景的错误处理不加
- 若能将 200 行减到 50 行，重写

### 3. Surgical Changes
**只改必须改。只清理自己的烂摊子。**

- 不"改进"相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格，即使与你做法不同
- 删除 own 改动导致未用的 import/变量/函数
- 每行改动应直接追溯到用户请求

### 4. Goal-Driven Execution
**定义成功标准。循环直到验证。**

- 任务转化为可验证目标
- 多步任务简述计划：`1. [步骤] → verify: [检查]`
- 强成功标准 → 可独立循环
- 不局限于单一功能测试，而是模块，甚至项目级别的端到端测试
- **编译器验证**: 编译产出 → hex → CPU 执行 → tohost 读出正确值

### 5. Requirements Integrity（需求完整性）
**需求完整性优先于实现简化。所有需求裁剪必须用户明确 approval。**

## 二、务实编码原则（代码质量）

### 十大铁律

#### 1. 命名即文档
- 名称揭示意图，非实现细节
- 避免缩写、单字母变量
- 使用领域语言（AST 节点类型：`FuncDef`, `BinaryOp`, `Compound`）

#### 2. 函数单一职责
- 理想 < 20 行，不宜超过一屏
- 只做一件事：一个解析函数对应一个语法产生式

#### 3. DRY & 正交性
- 编译器各阶段正交：Lexer → Parser → Sema → CodeGen
- 三次法则：复制两次后，第三次必须抽象

#### 4. 显式胜于隐式
- 寄存器分配显式记录在栈帧布局
- 符号表作用域显式栈结构

#### 5. 健壮边界
- 各阶段通过定义的接口通信：Token 流、AST、汇编文本

#### 6. 可测试设计
- **TDD 铁律**: 先写编译测试，再写编译器实现
- 每个阶段可独立测试：Lexer 输出 Token、Parser 输出 AST

#### 7. 尽早重构
- 发现坏味道立即小步重构

#### 8. 务实破窗
- 先实现端到端最小可用（阶段 1: `int main() { return 42; }`），再完善

#### 9. 自动化检查
- 编译产出必须通过 CPU TestBench 验证

#### 10. 注释解释意图
- AST 节点注释解释语义，非结构

## 三、编译器特有约束

### 编译阶段验证铁律

```
1. 词法分析器测试 → 验证 Token 流正确
2. 语法分析器测试 → 验证 AST 结构正确
3. 代码生成测试 → 验证汇编文本正确
4. 汇编器测试 → 验证机器码编码正确
5. 链接器测试 → 验证最终二进制正确
6. CPU 执行测试 → 验证 tohost 读出正确值
```

### 错误处理原则

- 精确定位：文件名 + 行号 + 列号
- 错误恢复：继续编译报告更多错误
- 三级消息：error / warning / info

### 内存安全

- AST 节点统一分配/释放
- 符号表生命周期明确
- 无内存泄漏容忍

## 四、核心执行约束（8 条）

```
1. 不探索清楚不实现
2. 不计划清楚不实现
3. 不完整覆盖需求不实现
4. 不测试通过不提交
5. 不验证成功不声明
6. 三次失败必须反思
7. 不见见证不变更（TDD Iron Law）
8. 不见场景缺口不进设计（BDD 智能缺口）
```

## 五、TDD 方法论（测试驱动开发）

### Iron Law（核心铁律）

```
NO CHANGE WITHOUT TEST WITNESS

编译器开发场景：
  New Feature：测试定义编译期望 → RED → 实现 → GREEN → CPU 验证
  Bug Fix：测试复现错误汇编 → RED → 修复 → GREEN → CPU 验证
  Refactor：测试记录编译行为 → GREEN → 重构 → 保持 GREEN
```

### 编译器测试金字塔

```
┌─────────────────────────────────┐
│  CPU 执行测试（tohost 验证）    │  ← 最终验收
├─────────────────────────────────┤
│  链接器测试（多文件链接）        │
├─────────────────────────────────┤
│  汇编器测试（指令编码）          │
├─────────────────────────────────┤
│  CodeGen 测试（汇编输出）        │
├─────────────────────────────────┤
│  Parser 测试（AST 结构）         │
├─────────────────────────────────┤
│  Lexer 测试（Token 流）          │  ← 基础验证
└─────────────────────────────────┘
```

## 检查清单

每次提交前确认：

- [ ] 新功能有对应测试
- [ ] 测试通过 CPU TestBench 验证
- [ ] 错误信息精确定位源码位置
- [ ] 无内存泄漏
- [ ] AST/Symbol 结构正确
- [ ] 汇编输出符合 RV32I ABI
- [ ] 代码比来时更干净

## Red Flags

```
❌ 无测试直接变更编译器代码 → Iron Law 违规
❌ 编译产出未通过 CPU 验证 → Goal-Driven 违规
❌ 添加未要求的优化 → Karpathy 违规
❌ 错误信息缺少位置 → 错误处理违规
```
