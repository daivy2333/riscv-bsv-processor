# 架构决策记录 (ADR)

> 记录关键架构决策及其背景

## 决策列表

### 2026-05-12 - 项目初始化

- **决策**: 采用十阶段渐进式开发，从 `int main() { return 42; }` 开始
- **原因**: 端到端验证优先，先跑通再扩展
- **影响**: 每阶段独立可验证，降低开发风险
- **替代方案**: 模块并行开发（风险高，难以集成）

### 2026-05-12 - 编译器流水线架构

- **决策**: 五阶段流水线：Preprocessor → Lexer → Parser → Sema → CodeGen → Assembler → Linker
- **原因**: 正交设计，各阶段职责清晰，便于独立测试和调试
- **影响**: 每阶段可独立验证，错误定位精确
- **替代方案**: 单遍编译（不利于优化和错误报告）

### 2026-05-12 - 实现语言选择

- **决策**: 编译器自身用 C (Micro-C 子集) 编写
- **原因**: 自举的唯一选择——编译器必须能被自己编译
- **影响**: 源码受限在 Micro-C 子集内，需阶段 10 验证自举
- **替代方案**: 用其他语言开发（无法自举）

### 2026-05-12 - 递归下降解析器

- **决策**: 手写递归下降解析器，不用 yacc/bison
- **原因**: 零依赖，完全可控，每个产生式对应一个函数
- **影响**: 代码直观，易于理解和调试
- **替代方案**: LR 解析器生成器（引入依赖）

### 2026-05-12 - 寄存器分配策略

- **决策**: 三阶段演进：全栈溢出 → 线性扫描 → 图着色
- **原因**: 正确性优先，性能逐步优化
- **影响**: 阶段 1-2 使用栈溢出（简单可靠），阶段 3+ 引入寄存器分配
- **替代方案**: 一开始就用图着色（复杂度高）

### 2026-05-12 - 软件乘除法

- **决策**: 乘除法用软件例程实现（`__mulsi3`, `__divsi3` 等）
- **原因**: 目标 CPU 无 M 扩展
- **影响**: 编译器自动链接算术库，对用户透明
- **替代方案**: 禁止乘除法操作符（不现实）

### 2026-05-13 - 二遍汇编器

- **决策**: 汇编器从一遍升级到二遍（标签解析），支持全 RV32I 指令集
- **原因**: 运行时库需要分支/跳转指令，必须支持标签解析
- **影响**: 链接器简化（拼接 crt0+lib+用户代码，一次汇编完成），不再需要手动 jalr 补丁
- **替代方案**: 保留一遍汇编 + 链接器手动补丁（需为每个跨文件引用添加补丁逻辑）

### 2026-05-13 - 拼接式链接

- **决策**: 链接器将 crt0.s、lib.S、用户代码拼接为一个文本块，交给二遍汇编器一次处理
- **原因**: 二遍汇编器已能解析所有跨文件标签引用
- **影响**: 链接器代码量减少，取消硬编码偏移量计算
- **替代方案**: 维持链接器手动补丁（随跨文件引用增多而复杂化）

### 2026-05-13 - 栈帧布局（阶段2）

- **决策**: 每个局部变量分配 4 字节栈空间，帧大小 = 4(ra) + 4*变量数，16字节对齐
- **原因**: 简单可靠，适合阶段 1-2 的栈溢出寄存器策略
- **影响**: 表达式中间结果借用 sp+0 作为临时槽，局部变量从 sp+4 开始分配
- **替代方案**: 寄存器分配（留待阶段3+引入）

### 2026-05-13 - 表达式左结合平级优先级

- **决策**: 所有四则运算符 `+ - * /` 同级，左结合
- **原因**: 实现简单，对验证目标（无优先级相关测试）完全正确
- **影响**: 优先级区分（`*/` 高于 `+-`）留待阶段 3
- **替代方案**: 立即引入优先级分层（需要在 parse_term 添加 parse_factor 层级）

### 2026-05-13 - 三级表达式优先级（阶段3）

- **决策**: parse_term(`*/`) → parse_addsub(`+-`) → parse_expr(`<`) 三级递归下降层次
- **原因**: 修复阶段2平级优先级错误（`1+2*3` = 9），支持比较操作符
- **影响**: 表达式解析层次清晰，`1+2*3` 现在正确 = 7，`i<3` 返回 0/1 比较结果
- **替代方案**: Pratt 解析器（更通用但更复杂，不适合自举 Micro-C）

### 2026-05-13 - 控制流语法糖展开（阶段3）

- **决策**: for 循环解析时展开为 while + 赋值；++ 后自增展开为 `i = i + 1`
- **原因**: 最小化 AST 类型数量（仅需 AST_IF/AST_WHILE，无需 AST_FOR/AST_INC）
- **影响**: 代码生成只需处理 if/while 两种控制流节点
- **替代方案**: 新增 AST_FOR 和 AST_INC 节点（增加 AST 类型和代码生成复杂度）

### 2026-05-13 - 独立标签范围（阶段3）

- **决策**: 使用单调递增计数器生成全局唯一标签 `.L0`, `.L1`, ... 用于 if/while 分支
- **原因**: 避免嵌套控制流中标签冲突，简单可靠
- **影响**: 直接使用 GNU 汇编器 `.L` 本地标签命名约定
- **替代方案**: 上下文作用域标签（标签带嵌套深度前缀，过度设计）

### 2026-05-13 - 多级表达式溢出槽（阶段3）

- **决策**: 预留 4 个 4 字节表达式临时溢出槽（sp+0~sp+12），使用深度计数器跟踪嵌套级别
- **原因**: 三级表达式优先级需要多级溢出槽，避免内外层表达式溢出值互相覆盖
- **影响**: 局部变量偏移从 sp+4 改为 sp+16，帧大小增加 12 字节
- **替代方案**: 维持单溢出槽 + 寄存器保存（需管理调用约定中库函数对 t2/t3 的使用）

### 2026-05-13 - 函数调用 AST 与解析（阶段4）

- **决策**: 新增 AST_FUNC_CALL 节点，参数列表复用 AST_VAR_DECL 链，函数定义参数存于 fn->right
- **原因**: 最小化 AST 节点类型，参数作为特殊局部变量（初始化自 a0-a7），代码生成复用现有符号表和局部变量机制
- **影响**: 参数从 a0-a7 寄存器加载到栈偏移（sp+16+），body 局部变量紧随其后
- **替代方案**: 独立 AST_FUNC_PARAM 节点类型（增加节点数和代码生成分支）

### 2026-05-13 - 传出参数溢出区（阶段4）

- **决策**: 在栈帧顶部预留 32 字节传出参数溢出区（8 槽位），位于 sp+frame-36~sp+frame-5
- **原因**: 支持多参数函数调用（参数在寄存器计算后需暂存），避免与表达式 4 级溢出槽冲突
- **影响**: 帧大小增加 32 字节（从 +0 变为 +32 + 4(ra) = +36），saved ra 偏移不变（frame-4）
- **替代方案**: 使用 caller-saved 寄存器暂存参数（a0-a7 被乘法除法调用覆盖）

### 2026-05-13 - 先定义后引用与递归（阶段4）

- **决策**: 函数必须在其被调用之前定义（源文件顺序），递归函数在定义时即知自身名称
- **原因**: 避免前向声明所需的二遍解析或函数表，汇编器二遍标签解析自动处理递归调用
- **影响**: fib 必须在 main 之前定义，递归调用 jal ra, fib 在标签 fib: 内被汇编器解析
- **替代方案**: 收集所有函数名后再解析体（显著增加解析器复杂度）

### 2026-05-13 - 阶段规划重构（特性原子化）

- **决策**: 重新规划阶段 5-10，采用特性原子化策略：每阶段新增一个原子特性
- **原因**: 原规划阶段 5 验证目标"链表操作"依赖阶段 7 的结构体特性，存在依赖冲突；原子化避免交叉依赖和大型重构
- **影响**: 阶段 5→6→8→9→10 依赖链单向，每阶段验证标准独立自洽
- **替代方案**: 合并特性（阶段 5 引入指针+结构体，复杂度高）、调整验证标准（链表推迟到阶段 7，阶段 5 验证目标变弱）

### 2026-05-14 - char 类型与指针算术（阶段7）

- **决策**: 类型系统新增 TYPE_CHAR，指针算术根据 base_type 决定 shift 量
- **原因**: char* 解引用用 lb，指针算术 sizeof(char)=1 无 shift，int* sizeof(int)=4 shift 2
- **影响**: gen_expr(AST_DEREF) 检查类型生成 lb/lw，指针加法检查 ptr_base_type
- **替代方案**: 固定 shift 2（char* 指针错误）或引入 sizeof 表达式（过度设计）

### 2026-05-14 - 双段内存布局（阶段7）

- **决策**: 编译器输出 .text（代码）和 .data（全局变量+字符串）两段，紧接排列
- **原因**: 全局变量和字符串需要固定地址，与代码分离便于独立管理
- **影响**: 汇编器需处理 .data/.word/.byte/.align 指令，Pass 1 分配地址，Pass 2 输出数据
- **替代方案**: 单段混合布局（代码数据交织，难以维护）或预留固定区域（浪费空间）

### 2026-05-14 - la 伪指令地址计算（阶段7）

- **决策**: la 在 Pass 1 额外 addr+=4，Pass 2 编码时加上 BASE_ADDR (0x80000000)
- **原因**: la 展开为 lui+addi 占 8 字节，Pass 1 只 addr+=4 导致标签地址偏移；汇编器使用相对地址，但 la 需要绝对地址
- **影响**: Pass 1 检测 la 伪指令额外增加地址，Pass 2 编码时加上基地址
- **替代方案**: 预留空间计算（复杂且易错）或 linker 后处理地址（延迟错误发现）

### 2026-05-14 - 预处理器独立模块（阶段9）

- **决策**: 新增 src/preprocessor.c 作为独立模块，在 lexer 之前处理预处理指令
- **原因**: 分离预处理和词法分析职责，预处理器处理 #include/#define/#ifdef，lexer 只处理纯 C 代码
- **影响**: main.c 先调用 pp_process() 获取预处理后的文本，再 tokenize()
- **替代方案**: lexer 内处理预处理指令（职责混淆）

### 2026-05-14 - 递归文件包含处理（阶段9）

- **决策**: 使用 included_files 数组记录已处理文件，循环包含返回空内容（不报错）
- **原因**: 头文件相互包含是常见模式，返回空内容避免重复定义错误
- **影响**: #include 已包含的文件不会产生重复内容，但宏定义仍会生效
- **替代方案**: 报错停止（严格但可能导致合法代码失败）

### 2026-05-14 - 条件编译栈设计（阶段9）

- **决策**: 使用 CondStack 数组（MAX_COND_DEPTH=16）跟踪嵌套条件，is_active() 检查所有层级
- **原因**: 条件编译可以嵌套，需要栈结构跟踪每层的激活状态和 else_seen 标记
- **影响**: 文件结束时检查 cond_depth > 0 检测 unmatched #ifdef
- **替代方案**: 递归处理（复杂度高）或单层状态（不支持嵌套）

### 2026-05-14 - extern 声明跳过生成（阶段9）

- **决策**: parse_extern_decl() 返回 AST_NOOP，codegen 不生成任何代码
- **原因**: extern 声明只是声明外部链接，不产生定义，实际符号在其他文件中定义
- **影响**: extern int g_val; 不产生 .data 定义，依赖 linker 解析跨文件符号
- **替代方案**: extern 声明生成符号引用标记（需要更复杂的符号解析）

### 2026-05-14 - 函数原型解析（阶段9）

- **决策**: 新增 AST_FUNC_DECL 节点，parse_func_decl() 解析 int foo(); 不生成代码
- **原因**: 头文件声明函数原型，跨文件调用需要前向声明
- **影响**: int foo(); 不生成函数体，调用时 linker 解析符号
- **替代方案**: 要求函数必须定义（不支持头文件声明）

### 2026-05-14 - 多文件链接器（阶段9）

- **决策**: 新增 linker_link_multi() 接收 asm_files 数组，拼接所有文件后统一汇编
- **原因**: 二遍汇编器已能解析跨文件标签，一次汇编比多次汇编+链接更简单
- **影响**: 所有 .s 文件 + crt0 + lib 拼接为一个文本块，交给 asm_assemble()
- **替代方案**: 多次汇编 + 符号表链接（复杂度高）

## 模块划分

### 编译器核心模块

| 模块 | 文件 | 职责 |
|------|------|------|
| **入口** | `main.c` | 命令行参数、驱动各阶段 |

| **寄存器分配** | `regalloc.c` | 活跃分析、寄存器映射 |

| **链接器** | `linker.c` | 多目标文件 → 可执行二进制 |
| **类型系统** | `types.c` | 类型定义、兼容性判断 |
| **符号表** | `symbol.c` | 标识符信息管理 |
| **错误报告** | `error.c` | 精确定位、格式化输出 |

| 模块 | 文件 | 职责 |
|------|------|------|

## 数据结构设计

### AST 节点类型

```
Program         顶层程序
FuncDef         函数定义
VarDecl         变量声明
StructDef       结构体定义
Compound        复合语句
If/While/For    控制流
BinaryOp        二元运算
UnaryOp         一元运算
Call            函数调用
Ident           标识符引用
IntLit/StrLit   字面量
...
```

### 符号表结构

```
作用域栈: 全局 → 文件 → 函数 → 块
每个符号: 名称 + 类型 + 存储类别 + 偏移/地址
```

### 栈帧布局

```
高地址
  ┌──────────────┐
  │ 溢出参数      │
  ├──────────────┤ ← 调用前 sp
  │ ra           │
  │ s0 (fp)      │
  │ callee-saved │
  ├──────────────┤
  │ 局部变量     │
  │ 临时空间     │
  └──────────────┘ ← sp
低地址
```

## 开发阶段规划（特性原子化）

> 每阶段新增一个原子特性，验证标准独立自洽，依赖单向。

| 阶段 | 目标 | 新特性 | 验证标准 | 依赖 |
|------|------|--------|----------|------|
| 1 | 最小可运行 | 函数定义 + return | `return 42;` → tohost = 42 ✅ | — |
| 2 | 算术与变量 | 变量声明 + 四则运算 | 7 个算术测试 ✅ | 阶段 1 |
| 3 | 控制流 | if/while/for + 比较操作符 | for 循环计数 ✅ | 阶段 2 |
| 4 | 函数调用 | 多参数函数 + 递归 | fib(10) = 55 ✅ | 阶段 3 |
| 5 | 指针基础 | `int *p` + `&x` + `*p` | `swap(&a, &b)` 指针传参 ✅ | 阶段 4 |
| 6 | 数组 | `int arr[n]` + `arr[i]` + 地址计算 | `arr_sum(arr, 5)` 数组求和 ✅ | 阶段 5 |
| 7 | 全局变量与字符串 | 全局变量 + 字符串字面量 + char 类型 | 5 个全局/字符串测试 ✅ | 阶段 4 |
| 8 | 结构体 | `struct` + 成员访问 + `p->field` | 链表节点创建遍历 | 阶段 5 + 6 |
| 9 | 多文件与预处理器 | 多文件链接 + `#include/#define` | 跨文件调用 + 头文件 | 阶段 7 + 8 |
| 10 | 自举准备 | `typedef` + `enum` + `volatile` | 编译器编译自身 | 阶段 9 |

**依赖链**: 1 → 2 → 3 → 4 → 5 → 6 → 8 → 9 → 10（阶段 7 可与 5/6 并行）

**设计原则**:
- 每阶段新增一个原子特性，增量最小
- 验证标准独立自洽，不依赖后续阶段特性
- 依赖单向，避免交叉依赖导致的重构
