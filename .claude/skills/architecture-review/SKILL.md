# Architecture Review (通用架构审查)

## 执行约束

- 默认先做架构/契约审查，后做实现建议
- 默认先定义契约，再讨论优化
- 默认区分根因、止血方案、正式方案
- 如果信息不足，先指出缺失信息和风险，不要过早下结论
- 禁止在没有完成分析前直接推荐补丁作为主方案

---

## 核心原则

**先判断问题类型：**

```
实现 bug    → 代码没有按设计正确实现
契约 bug    → 模块/组件/服务/流程之间的输入/输出、时序、状态、错误语义没有定义清楚
架构 bug    → 职责边界、数据流、控制流、依赖方向存在根本问题
多因素叠加  → 表面是实现问题，实际由契约缺失和架构缺陷共同造成
```

**默认优先级：**
1. 正确性 > 性能 > 简洁
2. 最小正确方案 > 复杂但不确定的方案
3. 契约清晰 > 经验依赖
4. 先修根因 > 先修症状
5. 显式设计 > 隐式假设
6. 能证明正确 > 看起来大概率没问题

---

## 分析任务

请按以下顺序分析，不要跳过步骤直接给补丁。

### 1. 问题定性
- 实现 bug / 契约 bug / 架构 bug / 多因素叠加？
- 当前现象是根因，还是更深层问题的外在表现？
- 这个问题是局部的，还是系统性的？

### 2. 目标与约束
- 系统原本要达成什么？
- 受哪些约束？（性能、一致性、实时性、安全性、资源、成本、兼容性）
- 问题是否是"目标"和"约束"冲突的结果？

### 3. 职责边界
- 谁创建数据？
- 谁拥有状态？
- 谁修改状态？
- 谁消费结果？
- 谁负责重试、回滚、兜底、超时、告警？
- 是否存在职责重叠或边界模糊？

### 4. 契约 / 接口语义
- 输入是什么？输出是什么？触发条件是什么？
- 成功 / 失败的定义是什么？
- 数据何时有效？有效期多久？
- 是否允许重复、乱序、超时、取消？
- 是否存在隐藏假设：
  - 固定时序
  - 固定顺序
  - 单线程 / 单实例 / 单写者
  - 同步语义
  - 强一致
- 如果这些假设被打破，系统是否仍然正确？

### 5. 状态机与生命周期
- 有哪些状态？转移条件是什么？
- 错误、取消、超时、重试是否在状态机里有处理？
- 是否存在未定义状态、卡死状态、无法恢复状态？
- 生命周期边界：初始化 / 运行 / 失败 / 恢复 / 关闭 / 清理

### 6. 数据流与控制流
- 数据如何流动？控制权如何流动？
- 是否存在"数据到了但控制未同步"或"控制先走但数据未就绪"？
- 是否存在重复写、丢写、乱序、早提交、读旧值、状态漂移？

### 7. 延迟假设
- 设计是否依赖固定延迟、固定顺序或同步语义？
- 如果处理晚 1 次、2 次、N 次，系统是否仍正确？
- 是否把"性能优化路径"误当成了"正确性路径"？

### 8. 失败模式
- 系统如何失败？最常见路径？最危险但最隐蔽的路径？
- 失败后是：显式报错 / 静默失败 / 部分成功 / 数据损坏 / 状态不一致？
- 能否检测、隔离、恢复？

### 9. 观测性
- 能否观察到关键状态、事件、边界条件？
- 是否有足够日志、指标、trace、断言、测试来证明分析结论？
- 哪些信息缺失导致调试困难？

### 10. 演进与扩展性
- 当前修复是否会让未来迭代更难？
- 是否会让后续功能必须继续堆补丁？
- 方案是否依赖某个暂时成立的假设？
- 如果规模扩大、并发增加、模块拆分，方案是否仍成立？

---

## 输出格式

```text
## 问题定性
【实现 bug / 契约 bug / 架构 bug / 多因素叠加】

## 表面现象
（描述看到的问题，不等同于根因）

## 根因分析
（从职责边界、契约语义、状态机、数据流、控制流角度分析）

## 关键设计缺口
（当前系统缺失或定义不清的点）

## 最小正确方案
（优先保证正确性，哪怕牺牲性能）

## 演进方案
（最小方案稳定后如何优化）

## 临时止血方案
（进度紧急时的短期方案；必须说明副作用和退出条件）

## 为什么不推荐继续打补丁
（补丁的副作用 / 未来维护成本 / 扩展性问题）

## 验证建议
（如何证明修复是正确的，而不是巧合 work）

## 风险与取舍
（牺牲了什么，换来了什么）
```

---

## 禁止直接采用的思路

在没有完成上面分析前，**不要默认推荐**：

- 多加 if/else 特判
- 多 stall 一拍 / 多等一次 / 多 sleep 一下
- 调优优先级让当前 case 通过
- 硬编码周期数、顺序、重试次数、超时时间
- 在多个地方重复保存同一份状态
- 用缓存、旁路、队列、重试去掩盖契约不清
- 用"通常不会发生"作为正确性依据

这些只能作为：临时止血方案 / 已证明契约正确后的局部优化 / 有明确回退计划的短期措施。

## 检查清单

建议任何补丁前，请先确认：

- [ ] 目标是否清楚？
- [ ] 约束是否明确？
- [ ] 职责边界是否清晰？
- [ ] 输入/输出契约是否明确？
- [ ] 状态机是否完整？
- [ ] 成功/失败语义是否定义？
- [ ] 是否存在隐藏的同步/顺序/单线程假设？
- [ ] 数据有效期和提交条件是否明确？
- [ ] 是否考虑了异常、超时、重试、取消、并发？
- [ ] 能否被日志/指标/trace/测试验证？
- [ ] 修复方案能否推广到未来相似问题？
- [ ] 当前建议是在修根因，而不是修表象？

**如果以上任意一项不明确，请优先补契约、补边界、补状态机，而不是直接打补丁。**

---

## 一句话原则

**先分清是"实现错"还是"设计错"；在设计还没被证明正确之前，不要把补丁当成方案。**
